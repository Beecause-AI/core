The relationship is non-monotonic. The weak model can’t use the graph; the mid model over-uses it (huge speed, real accuracy loss); only the top model converts it into efficiency without losing accuracy — because it has the judgment to verify selectively rather than trust the summary blindly.

Where Opus+KG lost its 2 points

  • Q5 — one deep root cause missed. Code-only Opus surfaced a subtle real cause — pg connection-pool exhaustion from cancel.ts holding a dedicated LISTEN connection per turn. Opus+KG found the breaker / max-iterations / credential causes but stopped before that one. The KG oriented it onto the well-lit path and it dug one level less.
  • Q8 — a hedge instead of a verify. It answered latency/cost from the KG with 0 tools and self-flagged “did not re-verify exact paths/columns.” Correct, but it traded one verification for speed.
  • Everywhere else it matched code-only — including the two questions that broke the weaker KG agents (Q9 authorized-domains, Q4 provider files).

Verdict & recommendation

The Knowledge Graph is a real accelerator — but how much it helps, and whether it costs accuracy, depends on the model’s judgment:

• Top model (Opus): the sweet spot. −37% tools / −19% tokens at near-identical accuracy (50→48). It orients on the KG and still verifies the hard claims, reproducing the deep root causes.
• Mid model (Sonnet): fast but risky. −90% tools but −10 accuracy — it lets the KG replace reading code and invents/oversimplifies.
• Weak model (Haiku): little upside. −9% tools, −4 accuracy — can’t convert the map into shortcuts and hallucinates when it tries.

Recommendation: pair the KG with a top-tier model for diagnosis, and engineer the “orient-then-verify” discipline into the tools so even mid models get Opus-like behaviour: KG results should return concrete codeRefs (file:line) and the agent must open them before asserting a path, mechanism, or root cause — never accept a 0-tool answer on an RCA question. That keeps the −37% efficiency while closing the accuracy gap. (The KG still can’t surface facts that live outside code — e.g. the Firebase authorized-domains misconfig was only inferable because the rebrand context was in scope — which is what the deferred telemetry/observability connectors would add.)

Method: 10 questions × 2 conditions × 3 models (Claude Haiku / Sonnet / Opus), all with Read/Grep/Glob/Bash on the repo; the +KG condition also receives the project knowledge-graph context. Speed = tool-call count; effort = subagent output tokens; accuracy graded blind by an Opus judge against ground truth from the source. Caveats: wall-clock excluded (parallel contention); one run per cell, n=10 (the n=6 Sonnet row used an earlier question set), so figures are directional, not definitive.
