RunAccuracy (code-only → +KG)Tool calls (→ +KG)KG effect

Weak model (n=10, this run) 46/50 → 42/50141 → 128 (−9%) Wash-to-negative — over-trusted the KG, skipped verification

A strong model uses the KG the right way — orient, then verify — so it reaches the same correct answer with far less code-spelunking. The weak model can’t reliably do that: it either ignores the KG and greps anyway (little speedup) or over-trusts the summary and stops verifying (accuracy loss).

What happened on incidents

  • On incidents the KG cut effort but not reliably (−16% tools, −10% tokens) — and lost accuracy (29 → 25). The weak model often read the KG and still grepped (Q6/Q9/Q10 used more tools), so the orientation rarely paid off; where it did shortcut, it sometimes shortcut into a wrong answer.
  • Failure mode — fabrication via shortcut (Q5). Given the KG, the weak model answered the “assistant intermittently fails” incident with 0 code reads and invented two non-existent paths (/packages/ai/, /packages/engine/worker). Code-only scored 5/5 on the same question by actually reading loop.ts, breaker.ts, message_queue, model_invocations.
  • Failure mode — confident wrong fact (Q1). With the KG the weak model asserted “web is deployed to Cloudflare Workers” (it’s Firebase Hosting) — a plausible-but-wrong specific it didn’t verify.
  • Where the KG genuinely helped: structural framing — it filled gaps the raw-code reader missed (Q2 web mirror-types, Q4 bootstrap registration), nudging those 4 → 5.
  • Both missed the real super-login root cause (Q9): neither named the actual incident — the serving domain missing from Firebase Auth authorized domains — because that fact lives in Firebase config, not the code or the KG. (A telemetry/observability-connector KG layer would close this.)

Takeaways

The Knowledge Graph is a force-multiplier for a capable agent and a liability for a weak one when used unguarded. The summary tempts a weak model to trust it over the filesystem — fast, but it manufactures plausible hallucinations (wrong paths, wrong deployment facts).

Recommendation — KG-for-orientation, code-for-truth:
• Use the KG to narrow the search, but require code verification before asserting any file path, deployment fact, or root cause.
• For weaker/cheaper models, make the KG tools return node ids → real file paths the agent must open (don’t let it free-narrate from digests), and prefer pairing the KG with a strong model for diagnosis.
• Keep the grounding discipline already shipped (no invented telemetry) — and extend it: have KG answers cite the concrete codeRef so any claim is checkable.

Method: 10 questions, two agents (same model — Claude Haiku — both with Read/Grep/Glob/Bash on the repo); only the code+KG agent also receives the project’s knowledge-graph context. Speed = tool-call count; effort = subagent output tokens; accuracy graded blind by a separate strong judge against ground truth from the source. Caveats: wall-clock excluded (parallel contention); n=10 single run, so directional not definitive; the strong-model comparison used a different 6-question set, so cross-run numbers show direction of the KG effect, not a like-for-like delta.
