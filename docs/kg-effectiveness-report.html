What the numbers say

  • Accuracy is at parity — both near-perfect (29 vs 30/30). With full code access, a strong model already answers these well; the KG didn’t raise the ceiling much. Its one edge: the code-only agent made a confident wrong claim (“graph-builder clones repos” — it actually reads via the GitHub API); the KG agent didn’t.
  • The KG roughly halves the investigation effort (41 → 20 tool calls). It’s the clear win: the agent reaches the same correct answer with far less code-spelunking. Biggest savings on the structural questions the graph is built for — architecture (−5), blast-radius (−4), external deps (−5).
  • The telemetry “trap” held. After the grounding fix, both agents correctly identified an OpenTelemetry-only stack and explicitly noted the Prometheus/Datadog/Sentry strings exist only inside the KG’s own detector code — no hallucination.
  • Where the KG adds little: a question already pinned to one file (approval, Q2) is a tie — there’s nothing to “navigate.”

Interpretation

For a strong model with full code access on a repo it can fully explore, the Knowledge Graph’s benefit shows up as efficiency and reliability rather than a big accuracy jump: ~half the exploration steps to the same answer, plus it steered the agent away from one fabricated detail. The advantage compounds exactly where you’d expect it to matter most in production:

• Larger / unfamiliar codebases — code-only exploration cost grows with repo size; the KG stays flat.
• Weaker / cheaper models — less able to reconstruct architecture from raw files; the KG hands it to them.
• Latency- or token-budget-sensitive RCA — fewer tool calls = faster, cheaper answers.
• Cross-repo questions — the KG already stitches multiple repos into one system map.

Method: 6 questions, two agents (identical model, both with Read/Grep/Glob/Bash on the repo); the only difference is the code+KG agent also receives the project’s knowledge-graph context. Speed measured as tool-call count (contention-free); accuracy graded blind by a separate judge against ground truth derived from the source. Caveat: wall-clock time was inconclusive this run (parallel execution + model “thinking” outliers swamped the signal), so tool-call count is the reported speed metric. Sample size is small (n=6); treat as directional.
